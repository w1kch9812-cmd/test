# Supply Chain Provenance And Deploy Gate Runbook

## Purpose

Production deploy candidates must be traceable to an approved workflow, a
specific subject digest, a generated SBOM, and a GitHub Artifact Attestations
record. A candidate without this evidence is not deployable, even if tests pass.

Policy SSOT:
`docs/architecture/platform-integration/supply-chain-policy.v1.json`

Approved workflow:
`.github/workflows/ci.yml`

Approved job:
`supply-chain-provenance`

Admission workflow:
`.github/workflows/production-deploy-admission.yml`

## Required Evidence

Each production candidate must have:

- SLSA build provenance generated by GitHub Artifact Attestations.
- A CycloneDX SBOM for the Node workspace candidate.
- A CycloneDX SBOM for the Rust API candidate.
- A subject digest matching the artifact selected for deploy.
- A CI run from the approved workflow on `refs/heads/main`.
- Load-test capacity evidence from an approved perf/staging run.
- A production edge admission check proving the generated traffic/auth WAF
  policy is wired into the deployment path.

Current subjects:

- `target/supply-chain/gongzzang-web-next-build.tgz`
- `target/release/api`

Current SBOM files:

- `target/supply-chain/gongzzang-node-workspace-sbom.cdx.json`
- `target/supply-chain/gongzzang-rust-workspace-sbom.cdx.json`

## Verification

Before promotion, download the candidate artifact from the approved workflow
run and verify provenance:

```bash
gh workflow run "Production Deploy Admission" \
  -f run-id=<attested-ci-run-id> \
  -f load-evidence-run-id=<load-test-workflow-run-id> \
  -f artifact-name=supply-chain-release-candidates
```

The admission workflow downloads the release candidate artifact from the
approved CI run, verifies provenance for both production subjects, verifies SBOM
attestations, downloads load-test capacity evidence from the approved load-test
workflow run, runs `scripts/ci/verify-load-test-capacity-evidence.ps1`, runs
production edge admission, and binds the gate to the `production` GitHub
Environment.

The load-test capacity evidence gate requires healthy evidence for
`api-read-mix`, `map-marker-mix`, and `platform-core-events`. Each accepted run
must use `baseline`, `stress`, `spike`, or `soak` from `perf` or `staging`.
`perf` evidence must target `perf.gongzzang.internal`; `staging` evidence must
target `staging.gongzzang.internal`. It rejects local/ci smoke evidence,
production targets, missing k6 summaries, non-zero k6 exit codes, and SLO
breaches.

For regional ingress, set the production environment variable
`GONGZZANG_WAF_REGIONAL_RESOURCE_ARN` to the ALB/API Gateway ARN that should
receive the generated WebACL. The Pulumi program reads that value through the
`wafRegionalResourceArn` config key and creates the `aws.wafv2.WebAclAssociation`.
The admission script rejects production deploys when the ARN is missing,
malformed, wildcarded, or not a supported WAFv2 regional association target. It
also runs a local Pulumi association preview and requires the
`regional_association=planned` evidence line, proving that the deployment program
plans a `WebAclAssociation` for the provided target ARN before promotion.

CloudFront production ingress is intentionally blocked by the same admission
script until the generated WebACL ARN is wired into the CloudFront distribution
module. Do not attach WAF manually in the AWS console.

Manual verification uses the same script:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass `
  -File scripts/ci/verify-production-deploy-candidate.ps1 `
  -ArtifactPath target/supply-chain/gongzzang-web-next-build.tgz `
  -Repository <owner>/gongzzang

powershell -NoProfile -ExecutionPolicy Bypass `
  -File scripts/ci/verify-production-deploy-candidate.ps1 `
  -ArtifactPath target/release/api `
  -Repository <owner>/gongzzang
```

The script rejects artifacts not built by the approved workflow or approved
source ref. Manual verification commands are:

```bash
gh attestation verify target/supply-chain/gongzzang-web-next-build.tgz \
  -R <owner>/gongzzang

gh attestation verify target/release/api \
  -R <owner>/gongzzang
```

Then verify the SBOM attestations:

```bash
gh attestation verify target/supply-chain/gongzzang-web-next-build.tgz \
  -R <owner>/gongzzang \
  --predicate-type https://cyclonedx.org/bom

gh attestation verify target/release/api \
  -R <owner>/gongzzang \
  --predicate-type https://cyclonedx.org/bom
```

The deploy must stop if:

- The artifact came from an unapproved workflow.
- The artifact was not built on `refs/heads/main`.
- The subject digest does not match the candidate being deployed.
- The artifact has no provenance attestation.
- The artifact has no SBOM attestation.
- Load-test capacity evidence is missing, local-only, smoke-only, unhealthy, or
  produced against a production target.
- The image or package reference uses a mutable tag without a digest.
- The production edge admission script fails.
- `GONGZZANG_WAF_REGIONAL_RESOURCE_ARN` is missing for regional ingress.
- The generated WebACL is not attached through Pulumi via `wafRegionalResourceArn`.

## Incident Handling

If verification fails, treat the candidate as untrusted. Rebuild from the
approved workflow and attach the failed run URL to the release record. Do not
manually repackage or re-upload the artifact outside the approved workflow.
